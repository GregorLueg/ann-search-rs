## Standard indices benchmarks and parameter gridsearch

Below are all of the standard indices shown and their performance. These
are run via:

```bash
# Run with default parameters
cargo run --example gridsearch_<INDEX> --release -- --distance euclidean

# Override specific parameters
cargo run --example gridsearch_<INDEX> --release -- --distance cosine

# Available parameters with their defaults:
# --n-samples 150_000
# --dim 32
# --n-clusters 30
# --k 15
# --seed 42
# --distance cosine
# --data gaussian
```

To note, the data sets are just there for testing and get an idea on the
behaviour with different types of structure in the data. You should not
extrapolate from the results here what the ideal parameters are for your
specific problem. Also, this is a single run. There is some variation in terms
of exact times, but the overall trends should reproduce on your system with
your specific specifications.

### Annoy

`Approximate nearest neighbours Oh Yeah`. A tree-based method for vector
searches. Fast index building and good query speed. The size of the index
however increases quite drastically the increased number of trees. This version
does not support the on-disk querying of the original and is designed as a
pure in-memory index.

**Key parameters:**

- *Number of trees (nt)*: The number of trees to generate in the forest
- *Search budget (s)*: The search budget per tree. If set to auto it uses
  `k * n_trees * 20`; versions with a `10x` or `5x` (i.e., less) are also shown.

Self is queried with the default search budget.

### BallTree

A metric tree data structure that partitions data points into nested
hyperspheres. Unlike Annoy, which relies on random projections, BallTree uses
the triangle inequality to prune the search space efficiently. This means
instead of a forest it can just use a single tree.

**Key parameters:**

* *Search budget (s)*: The percentage of the total nodes/points to inspect
  during the query. If not provided, it will default to `5%`.

Self is queried with the default search budget.

### HNSW

Hierarchical navigatable small worlds. A graph-based index that needs more time
to build the index. However, fast query speed and compared to some other
approximate indices limited memory finger print.

**Key parameters:**

- *M (m)*: The number of connections between layers
- *EF construction (ef)*: The budget to generate good connections during
  construction of the index.
- *EF search (s)*: The budget for the search queries.

Self is queried with `s=100`.

Inverted file index. Uses Voronoi cells to sub-partition the original data.

  Numbers here tend to be `sqrt(nlist)` or up to 5% of the nlist.

### Kd forest

A forest of randomised Kd-trees for approximate nearest neighbour search. Each
tree splits along coordinate axes selected randomly from the top highest-spread
dimensions, providing diversity across trees. Split decisions are cheap (single
coordinate comparison rather than a full dot product), which makes tree
traversal fast and build times low. For Cosine distance, trees are built on
normalised vectors so that axis-aligned splits approximate angular partitions.
Kd-trees degrade in very high dimensions but remain competitive for approximate
search up to ~100-200 dimensions when combined with a forest.

**Key parameters:**

- *Number of trees (nt)*: The number of trees to generate in the forest
- *Search budget (s)*: The search budget per tree. If set to auto it uses
  `k * n_trees * 20`; versions with a `10x` or `5x` (i.e., less) are also shown.

Self is queried with the default search budget.

### LSH

Locality sensitive hashing. Can be a very fast index, but with increasing
tables, the memory fingerprint does increase.

**Key parameters:**

- *Number of tables (nt)*: The number of independent hash tables to generate.
  More tables improve recall at the cost of query time and memory.
- *Number of bits (nb)*: The bit resolution of the hash functions. Higher values
  create finer partitions but may reduce collision rates.
- *Number probes (n)*: How many additional buckets to explore per given
  hash table. A good heuristic tends to be `number of bits / 2` to
  `number of bits`.
- *Max candidates (s)*: The search budget limiting the number of candidates
  examined. Set to 'auto' for full search or a fixed value (e.g., 5k) for faster
  queries with lower recall.

### NNDescent

The NNDescent implementation in this crate, heavily inspired by the amazing
[PyNNDescent](https://github.com/lmcinnes/pynndescent), shows a very good
compromise between index building and fast querying. It's a great arounder
that reaches easily performance of ≥0.98 Recalls@k neighbours. You can even
heavily short cut the initialisation of the index with only 12 trees (instead
of 32) and get in 4 seconds to a recall ≥0.9 (compared to 48 seconds for
exhaustive search)!

**Key parameters:**

- *Number of trees (nt)*: Number of trees to use for the initialisation. If set
  to auto, it defaults to `min(5 + round(n^0.25), 32)`
- *Search budget (s)*: The search budget for the exploration of the graph during
  querying. Here it defaults to `k * n_trees`.
- *Diversify probability (dp)*: This is based on the original papers leveraging
  NNDescent and it is supposed to remove redundant edges from the graph to
  increase query speed.

### Vamana

`Disk Approximate Nearest Neighbors`. A graph-based method for vector searches
that powers DiskANN. Builds a directed navigable graph initialised with random
long-range edges, then refines it over two passes using a beam search and a
robust pruning heuristic (alpha-pruning). The entry point is the dataset medoid.
This is the in-memory variant.

**Key parameters:**

- *Max degree (R)*: Maximum out-degree per node in the graph
- *Build list size (L_build)*: Beam width during index construction
- *Search list size (ef_search)*: Candidate list size at query time. Defaults
  to `75` if not set.

Self is queried with `ef_search=75` (default value).

## Conclusions

In terms of choice of the index, it really depends on your use case and the
constraints you are having. The tree-based version tend to work well in typical
small data cases with ≤500,000 samples and not too much dimensionality. They
build fast and can query fast. At higher dimensions, they are starting to be
less efficient. LSH is overall one of the weakest indices here. Build times are
very fast, but the data structure needs to be right for this index to perform
well. IVF is excellent and it is not surprising that it is often used and
can also be very easily combined with quantisiers and due to the flat
querying pattern it also very amenable to GPU acceleration. The graph-based
ones are all excellet. Vamana and HNSW can take longer when increasing the
edge numbers to build for more precision. If you are more interested in high
fidelity queries with high speed, these are great choices. The NNDescent
version provided in the crate was designed to be fast in index building
